> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship the seven approved v0.3 features for datasette-assignments in priority order: live copy-editing, public gallery, append tasks, consensus views, share prompt, drag-reorder + list progress, debt cleanup.

**Architecture:** All work lives in `plugins/datasette-assignments/` (views/templates/static/schema) plus the generated-app template. **Tech Stack:** Same as the plugin: Datasette 1.0a35 (pinned), vanilla JS/CSS, Jinja, pytest.

### Task 1: Edit live assignments (copy edits only)

**Files:**
- Modify: `datasette_assignments/views.py` (new handlers `assignments_edit`), `__init__.py` (route), `templates/assignments_new.html` (edit-mode conditionals), `static/builder.js` (edit-mode locks), `templates/assignments_manage.html` (Edit link), `datasette_assignments/schema.py` (add `merge_editable`)
- Test: `tests/test_edit.py` (new), `tests/test_schema.py` (append merge tests)

**Interfaces:**
- Produces: `merge_editable(stored: dict, posted: dict) -> dict` in schema.py — returns a NEW definition equal to `stored` with only these accepted from `posted`: top-level `name`, `instructions`; per matching field (matched by position AND id — mismatch in count/ids/kinds/types raises `DefinitionError`): `label`, `help`, header/paragraph `text`; for select/checkbox_group: posted `options` must start with the stored options in order (only appends accepted; anything else raises). Result re-validated with `validate_definition`.
- Routes: `GET /-/assignments/<slug>/edit` (owner-or-root; renders the studio template with `edit_mode=True`, `definition_json` = stored definition, and `hand_edited` flag) and POST same path (fields `definition`, `csrftoken`, and `confirm_overwrite` required when hand-edited) → merge, validate, `registry` update (add `registry.update_definition(datasette, slug, defn)`), `AppsRegistry.update_stored_app(app_id, name, description[:200], render_app_html(defn, db_name), actor_id=...)`, redirect to manage.
- `hand_edited` detection: current app HTML (from apps registry current revision) != `render_app_html(stored_defn, db_name)`.
- Edit-mode client: template hides palette/CSV/mode/advanced-slug sections (`{% if not edit_mode %}`); builder.js in edit mode (flag via `window.__editMode`) disables remove/move/drag, renders id/type as static text, makes required/gallery/companion checkboxes disabled, existing options read-only + Add-option still active, and posts to the edit URL. Live preview still works.

**Merge-function tests (write in test_schema.py, TDD):**
```python
def test_merge_editable_accepts_copy_changes():
    stored = validate_definition(make_defn(slug="tips"))
    posted = json.loads(json.dumps(stored))
    posted["name"] = "New Name"
    posted["instructions"] = "New words"
    posted["fields"][0]["label"] = "Better label"
    posted["fields"][0]["help"] = "hint"
    merged = merge_editable(stored, posted)
    assert merged["name"] == "New Name"
    assert merged["fields"][0]["label"] == "Better label"


def test_merge_editable_rejects_structural_changes():
    stored = validate_definition(make_defn(slug="tips"))
    posted = json.loads(json.dumps(stored))
    posted["fields"][0]["type"] = "textarea"        # type change ignored
    posted["fields"][0]["required"] = not posted["fields"][0]["required"]
    merged = merge_editable(stored, posted)
    assert merged["fields"][0]["type"] == stored["fields"][0]["type"]
    assert merged["fields"][0]["required"] == stored["fields"][0]["required"]
    posted2 = json.loads(json.dumps(stored))
    del posted2["fields"][0]                        # field removal -> error
    with pytest.raises(DefinitionError):
        merge_editable(stored, posted2)


def test_merge_editable_options_append_only():
    stored = validate_definition(make_defn(slug="sel", fields=[
        {"kind": "input", "type": "select", "id": "pick", "label": "Pick",
         "help": "", "required": False, "gallery": False,
         "missing_companion": False, "options": ["a", "b"]}]))
    posted = json.loads(json.dumps(stored))
    posted["fields"][0]["options"] = ["a", "b", "c"]
    assert merge_editable(stored, posted)["fields"][0]["options"] == ["a", "b", "c"]
    posted["fields"][0]["options"] = ["a", "x", "c"]   # rename -> error
    with pytest.raises(DefinitionError):
        merge_editable(stored, posted)
```

### Task 4: Consensus/agreement views

**Files:**
- Modify: `datasette_assignments/schema.py` (`build_ddl`, `drop_ddl`), `datasette_assignments/__init__.py` (permission hook covers `a_%_agreement`)
- Test: `tests/test_schema.py` (append), `tests/test_permissions_views.py` (append)

`build_ddl` (tasks mode only) additionally emits:
```sql
CREATE VIEW a_<slug>_agreement AS
SELECT t.id AS task_id, t.<title_col> AS task_title,
  (SELECT COUNT(*) FROM a_<slug>_responses r WHERE r.task_id = t.id) AS response_count,
  -- per input-field column col:
  (SELECT r.col FROM a_<slug>_responses r WHERE r.task_id = t.id
    GROUP BY r.col ORDER BY COUNT(*) DESC, r.col LIMIT 1) AS col_majority,
  (SELECT COUNT(DISTINCT r.col) FROM a_<slug>_responses r WHERE r.task_id = t.id) AS col_distinct
FROM a_<slug>_tasks t
```
(title_col = task_title_column or first task column; `_missing` companion columns are EXCLUDED from majority/distinct generation — only the primary field columns.) `drop_ddl` drops it. Hook: the view-table deny/allow SQL patterns extend from `a_%_responses`-only to also match `_agreement` (mirror however the current SQL matches — likely a LIKE per slug; add the second name per slug).

**Tests (TDD):** schema: agreement view exists for tasks mode, absent for form mode; majority math — 2 responses "x" + 1 "y" → majority "x", distinct 2; drop_ddl removes it. Permissions: anon + non-owner GET `/assignments_data/a_mayors_agreement.json` → 403; owner + root → 200.

Steps: RED → implement → GREEN → commit `feat(plugin): per-task agreement views (owner-only)` → push.
