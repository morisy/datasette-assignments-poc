# v0.3 Wave — Design

**Date:** 2026-07-15
**Status:** Approved by user ("copy edits only" + "plugin page" gallery chosen via Q&A; full design approved verbatim)
**Order is priority order.** Branch `v0-3-wave`; pushed checkpoint commit per item; deploy+smoke after items 2, 5, 7; tag `v0.3.0` at the end. All 40 existing tests stay green throughout; each feature adds its own. Existing live assignments keep working untouched.

## 1. Edit live assignments (copy edits only)

- Manage page gains "Edit assignment" → `GET|POST /-/assignments/<slug>/edit` (owner-or-root), reusing the studio template in **edit mode**.
- Edit mode locks structure: no palette, no add/remove/reorder of fields, ids/types/required/gallery/companion flags frozen and rendered read-only. Editable: assignment name, instructions, field labels, help text, header/paragraph text, and ADDING options to select/checkbox_group (existing options render read-only — collected data references them). Slug immutable. Live preview active in edit mode.
- Save: server merges the posted definition into the stored one **accepting only the editable properties** (never trust the client to respect locks); validates; updates registry definition JSON; updates the app via datasette-apps `update_stored_app` (new revision — old HTML preserved in revision history); updates app name/description. Stored queries and schema untouched (nothing editable affects them).
- Hand-edit warning: before rendering the edit form, compare the app's current HTML to `render_app_html(stored_defn)`. If different, show a warning banner ("This app's HTML was customized by hand; saving will overwrite those edits") and require a `confirm_overwrite` checkbox on save.

## 2. Gallery page

- Public `GET /-/assignments/<slug>/gallery` (no auth): assignment name, instructions, published-response count, one card per `is_public=1` response showing Gallery fields only + submitted date, newest first, paginated 50/page (`?page=N`), "Contribute →" button linking to the app path. Family styling (census palette), server-rendered plugin template `assignments_gallery.html`.
- 404 when the assignment doesn't exist OR has no Gallery fields. Empty state ("Nothing published yet") when gallery exists but no public rows.
- Manage page links to the gallery. Reads go through the plugin server-side (plugin code is not permission-gated); only Gallery-flagged columns of public rows are ever rendered — same exposure as the existing public view.

## 3. Append tasks

- Manage page, tasks mode only: CSV paste textarea + upload button (same client component as the builder) → `POST /-/assignments/<slug>/add-tasks` (owner-or-root). Server: parse CSV; sanitized headers must equal the assignment's `task_columns` exactly (order-insensitive; error names the mismatch); ≤10000 rows; insert via existing `creator.insert_tasks`; redirect back with "Added N tasks."

## 4. Consensus views (tasks mode)

- Creation additionally generates view `a_<slug>_agreement`: per task row — task id, title column value, `response_count`, and per input-field column: `<col>_majority` (most frequent value via correlated subquery) and `<col>_distinct` (COUNT DISTINCT). Generated by `schema.build_ddl`; dropped by `drop_ddl`.
- **Privacy:** majority values are response content → the permission hook's deny extends to `a_%_agreement` views exactly like `a_%_responses` tables (owner+root only). Tests must cover this.
- No backfill for pre-existing assignments (none in tasks mode exist in production).

## 5. Share prompt

- Generated app (app_template.html): the form-mode thank-you state and tasks-mode "all done"/"caught up" states gain a "Share this assignment" row with a copy-link button. URL: derived from `document.referrer` at runtime (the contributor's current page); when the assignment has gallery fields, offer the gallery path (`new URL('/-/assignments/<slug>/gallery', referrer)`) with copy + "See what's been found" link; otherwise copy the referrer itself (the app page). If referrer is unavailable/unparseable, the share row hides. Clipboard via `navigator.clipboard.writeText` with a select-text fallback; "Copied!" toast reuses the existing toast.
- Slug and has-gallery are render-time facts — bake them into the template constants.

## 6. Builder drag-reorder + list progress bars

- Field cards get a drag handle (HTML5 draggable; drop reorders the fields array and re-renders; `notifyChanged()` fires). Up/down buttons remain (keyboard accessibility). No dependency.
- `/-/assignments` list: per-assignment mini progress bar computed server-side — tasks mode: done/total tasks (+ % fill); form mode: response count with a neutral full-width bar treatment. Reuses the manage page's progress queries.

## 7. Debt cleanup (one commit)

- `validate_definition`: `responses_per_task` must be an integer ≥ 1 — `0`/negative/non-int now error (no silent default-to-3 for falsy-but-present values; absent still defaults to 3).
- `build_queries`: `next_task_<slug>` gains `AND (SELECT value FROM a_<slug>_config WHERE key='status') = 'open'`.
- Wizard/server validation: when `task_image_column` is set, at submit time no schema check is possible per-row (CSV values), but `validate_definition` documents CSP as the enforcement; ADD client+server note only — plus a `https?://` startswith check in the generated app before rendering the image (skip rendering otherwise). (CSP remains the real wall.)
